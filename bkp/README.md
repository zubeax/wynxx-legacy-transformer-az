# RfP Legacy Transformer Mission Statement

The *Legacy Transformer* feature of Wynxx is advertised as supporting the process of 
 modernizing legacy systems efficiently by:​

    - Extracting Business Rules: Identifies and extracts business logic and rules from existing programs or systems.​
    - Facilitating Rule Modification: Provides tools to review and modify extracted business rules before proceeding with system reconstruction.
    - ​Automated Rebuilding: Translates and rebuilds these rules into a modern target application language and architecture using automated code generation.​


The intention with the packages in this project is to
- build hands-on expertise with the Legacy Transformer
- create real-world use cases that can be used in client conversations
- create a set of regression tests to be used with future Legacy Transformer releases

The applications in this project were created as representative examples for the
documented use cases from the Excel Spreadsheet [`zOS Application Flavors`](https://git.gft.com/alze/rfp-legacy-transformer/-/blob/main/zOS%20Application%20Flavors.xlsx?ref_type=heads).
They are designed to be self-contained and ready to be built, installed and executed in an arbitrary z/OS environment.

Going forward each package will be migrated with the Wynxx Legacy Transformer to an equivalent Java Application. The results of the migration and subsequent test process will be documented for each application.

## Project Structure

```
./Legacy Use Cases                  # created either manually or generated by Copilot
./Use Cases migrated with Wynxx     # translated by the Wynxx Legacy Transformer
./Use Cases migrated with Copilot   # translated by copy/pasting into a Copilot chat
./Wynxx Archetypes                  # archetypes required by the Wynxx 'Code Generation' step
./Wynxx Documentation               # bits and pieces of stuff collected during the RfP
```

### Package Structure

```
./<package>                      # package root directory
           /src                  # cobol source code when no other languages are present
           /asm                  # assembler source code (optional)
           /cob                  # cobol source code
           /db                   # db/2 ddl (data definition language)
           /ims
               /ddl              # IMS PSB/ACB members
               /haldb            # JCL to create the IMS database
           /data                 # generated test data
           /jcl                  # build, install and execution job control
           /logs                 # output created either by the original or the migrated application
```
